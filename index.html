  1. merge작동원리
    • 가장 일방적인 방법 merge = 3-way merge

      메인브런치와 서브브런치 둘다 commit를 한다음 merge 할때

      선이 두개로 나뉘어서 합쳐짐

    • fast-foward merge

      메인 브런치에는 commit없고 서브 브런치에만 commit이 있을 경우

      merge할경우 자동으로됨, 서브브런치의 이름이 메인브런치로 바뀜

      강제로 3-way merge하는 방법 git merge --no-ff

  2. merge 하는 3가지
    • 일반적인 merge

      메인 브런치로 switch해서 marge 해야함

      git merge 서브브런치이름

    • rebase

      3-way merge 일경우 로그를 확인하면 서브브런치의 내용까지 나온다.
      간단한 브런치의 로그내용을 없앨 수가 있다. 쉽게 말해 fast-forward merge

      뒤로 갈수록 복잡해지는 marge 경로를 좀더 단순하게 표현해준다.(결국 view 문제)

      [실행방법]
      git switch 새로운브랜치 >> git rebase main
      git switch main >> git merge 새로운브랜치

      conflict가 많이 발생한다.

    • squah and merge

      서브 브런치의 commit 내용들을 메인 브런치로 commit 순간이동 해준다.

      메인브런치에서 git merge --squash 서브브런치이름33

  1. 브랜치로 협업하기
    • 로컬에서 브랜치만들고 업로드

      git branch 브랜치이름
      git switch 브랜치이름
      add . => commit
      git push origin 브랜치이름
      브랜치 이름을 안적으면 모든 브랜치가 다 업로드 됨

    • Pull request 하기

      협업하기 전에 대부분 코드리뷰/검토를 한다.

      git.hub사이트에서 pull requests 클릭

      merge 하기 >> 세가지방법이 있음
      rebase and merge 하면 개꿀

    • 브랜치 전략(git flow)

      브랜치 main
      말그대로 원본

      브랜치 develop
      main의 복사본
      실질적으로 여기서 개발하지 않느다.

      브랜치 feature
      develop의 복사본
      실질적 개발은 여기서 이루어진다.
      브랜치 이름을 feature/friend, feature/food 이런식으로 작성
      feature/friend가 완료되면 develop와 merge
      브랜치 feature/friedn안에 브랜치feature/food를 만들고 개발
      feature/food를 develop에 merge

      브랜치 release
      feature에서 개발 후 develop에 merge했으니 main에 바로 merge?
      상남자라면 가능!!!하지만 우리는 쫄보니깐 한번도 체크하자
      개발완성되어 merge한 develop를 한번더 브랜치 relase로 복사후
      수정할꺼 하고 그다음에 main과 merge하면 끝!!

      브랜치 hotfix
      첫번째 업데이트후 급한 버그가 생겼을 때 main를 브랜치 hotfix로 복사해서 버그 수정하기
      바로 main에 merge 하고 develop에도 merge 해주기
